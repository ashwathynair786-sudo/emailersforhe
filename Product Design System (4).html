Philosophy

The product is not the website. A landing page earns attention; an application respects it. These are the deliberate divergences from the marketing system — each one in service of an interface someone uses for eight hours, not eight seconds.

State colors are a shared tier, not a place to be clever. Error #DC2626, warning #F59E0B, success #059669 are identical to the marketing system — a user crossing from the website into the product must read the same green as "good" and the same red as "bad." Distinctiveness in product comes from component form (the signal-mark badge above), never from inventing bespoke state hues that fragment meaning. Info #2563EB is the one product-only addition — a clear blue, deliberately distinct from navy primary so "information" never reads as "action."

Buttons

Six variants, three sizes. Navy primary is the everyday action. Orange accent is rationed to the single most important action on a screen — the brand spark, not the default.

Sizes & icons

The one-accent rule: a screen has at most one orange button — "Publish assessment," "Send invites," "Push to ATS." Everything else (Save, Cancel, Filter, Edit) is navy primary, secondary, or ghost. If two things are orange, neither is special.

Badges & status

Status communication is the product's most frequent job. State colors do the talking; the score badge is the one place product borrows the brand's "big number" instinct.

The standard pattern — a soft tinted pill with darker same-hue text — ships with every component library, so it reads as generic. We invert it: the state color lives in a small square mark; the label stays neutral in mono caps. Chart strategy — three ramps

Categorical for nominal series (distinct hues), sequential for one measure by intensity (navy tints), diverging for above/below a midpoint (orange↔navy). Hue carries category; lightness carries magnitude — never mix the two jobs. Square legend marks, mono axis labels, dashed tick gridlines throughout. Score-band fills stay on the state colors.

Icons

The system uses Phosphor — a geometric, even-stroke family with six weights. Its regular weight matches the UI stroke; duotone gives an on-brand emphasis option (primary tone = navy, secondary = a faint navy wash). Icons inherit currentColor, so they pick up text and state colors automatically.

Do

  • Default to regular weight at 16–20px; let icons inherit currentColor.
  • Use duotone sparingly for section headers / empty states, primary tone in navy.
  • Use fill only for active/selected states (e.g. a selected nav item).
  • Pair an icon with a text label wherever space allows.

Don't

  • Don't mix Phosphor with another icon pack — stroke and grid won't match.
  • Don't color icons with lime, or use orange except as the one accent.
  • Don't use thin/light below 20px — strokes disappear at dense sizes.
  • Don't scale a 16px icon up to a hero size — step to a larger weight instead.

Tailwind config

Extend the theme so utilities like bg-accent, text-success, font-mono resolve to brand values. Fonts wired to Geist + JetBrains Mono.

// tailwind.config.ts
export default {
  theme: {
    extend: {
      colors: {
        background: "hsl(var(--background))",
        foreground: "hsl(var(--foreground))",
        primary: { DEFAULT: "hsl(var(--primary))", foreground: "hsl(var(--primary-foreground))" },
        accent: { DEFAULT: "hsl(var(--accent))", foreground: "hsl(var(--accent-foreground))" },
        success: "hsl(var(--success))",
        warning: "hsl(var(--warning))",
        info: "hsl(var(--info))",
        destructive: { DEFAULT: "hsl(var(--destructive))", foreground: "hsl(var(--destructive-foreground))" },
        border: "hsl(var(--border))",
        ring: "hsl(var(--ring))",
      },
      fontFamily: {
        sans: ["Geist", "system-ui", "sans-serif"],
        mono: ["JetBrains Mono", "ui-monospace", "monospace"],
      },
      fontSize: { base: ["14px", { lineHeight: "1.5" }] },
      borderRadius: {
        lg: "12px", md: "calc(var(--radius) + 2px)",
        DEFAULT: "var(--radius)", sm: "4px",
      },
    },
  },

Install the matching Shadcn components: npx shadcn@latest add button input select dropdown-menu table tabs dialog badge sonner. Variants map 1:1 — variant="default" = navy, add a custom variant="accent" for the orange key CTA.

Do / don't

The guardrails that keep product feeling like HackerEarth without feeling like the brochure.

Do

  • Use navy as the default action; ration orange to one key CTA per screen.
  • Set everything in Geist; reserve mono for numbers, IDs, code, timestamps.
  • Keep state colors meaning the same thing in every table, badge, and chart.
  • Lean on density — 14px base, 4px grid, 6px radii.
  • Let the data be the visual. Whitespace and alignment do the design work.
  • Map everything to Shadcn semantic tokens so theming stays centralized.
  • Use the score badge as the one sanctioned "brand number" moment.

Don't

  • Don't use Bricolage Grotesque anywhere in product — it's the marketing voice.
  • Don't bring contours, compass marks, or cartographic tickmarks into the app.
  • Don't use the warm cream canvas — product runs cool near-white.
  • Don't make orange the default button; hierarchy collapses if everything pops.
  • Don't use lime for actions — it's a data-highlight color only.
  • Don't introduce hex values outside the token set; extend tokens instead.
  • Don't inflate spacing to marketing scale — product earns its density.
